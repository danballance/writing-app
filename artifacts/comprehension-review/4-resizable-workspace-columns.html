Resizable workspace columns

This change turns the desktop workspace from fixed Tailwind grid tracks into a CSS-variable layout that users can resize, reset, and carry across sessions. The affected path is the three-column desktop shell: project navigation, editor, and writing partner.

The route through the new behavior

The visible behavior is narrow but touches layout, persistence, pointer input, keyboard input, and accessibility metadata. On desktop widths only, each side column now owns a resize separator. Dragging or using the keyboard writes a numeric width into React state. App converts that state into CSS custom properties on the grid.

Navigation columnResizable from the right edge. Stored under scribe-navigation-column-width.
Editor columnNo handle of its own. Its minimum usable width is protected by maximum side-column calculations.
Writing partner columnResizable from the left edge. Stored under scribe-context-column-width.
const MIN_NAVIGATION_WIDTH = 220;
const MAX_NAVIGATION_WIDTH = 380;
const MIN_CONTEXT_WIDTH = 280;
const MAX_CONTEXT_WIDTH = 720;
const MIN_EDITOR_WIDTH = 520;
const NAVIGATION_WIDTH_KEY = "scribe-navigation-column-width";
const CONTEXT_WIDTH_KEY = "scribe-context-column-width";

Those constants define the main invariant: side columns may move, but they should not compress the editor below 520px. The maximum for one side depends on the current measured width of the other side.

Where the chosen widths live

Widths are stored as numbers in localStorage. Loading is defensive: invalid, missing, out-of-range, or inaccessible storage returns null, which means "use CSS defaults." Saving follows the same fault-tolerant pattern.

function readSavedWidth(key: string, min: number, max: number) {
  try {
    const width = Number(window.localStorage.getItem(key));
    return Number.isFinite(width) && width >= min && width <= max ? width : null;
  } catch {
    return null;
  }
}

function saveWidth(key: string, width: number | null) {
  try {
    if (width === null) {
      window.localStorage.removeItem(key);
    } else {
      window.localStorage.setItem(key, String(width));
    }
  } catch {
    // Column resizing remains available when storage is blocked.
  }
}

Maintenance note: null is meaningful. It does not mean zero width or unknown width. It means the CSS should decide the column size. The reset handlers depend on that distinction.

The editor is protected by dynamic max widths

The side panels do not have fixed maximums alone. Each maximum also accounts for the workspace width, the opposite column, and the minimum editor width. This is the part that prevents a wide writing partner panel and a wide navigation panel from squeezing the editor too far.

const getMaximumNavigationWidth = useCallback(() => {
  const workspaceWidth =
    workspaceRef.current?.getBoundingClientRect().width ?? window.innerWidth;
  const contextWidth =
    contextColumnRef.current?.getBoundingClientRect().width ?? MIN_CONTEXT_WIDTH;
  return Math.max(
    MIN_NAVIGATION_WIDTH,
    Math.min(
      MAX_NAVIGATION_WIDTH,
      workspaceWidth - contextWidth - MIN_EDITOR_WIDTH,
    ),
  );
}, []);

The context calculation mirrors this, using the navigation column width. Both functions allow maxWidth to shrink when the viewport shrinks or when the other side panel has already claimed space.

useEffect(() => {
  const constrainSavedWidths = () => {
    if (!window.matchMedia("(min-width: 1280px)").matches) {
      return;
    }

    setNavigationColumnWidth((width) => {
      if (width === null) {
        return null;
      }
      const constrained = Math.min(width, getMaximumNavigationWidth());
      saveWidth(NAVIGATION_WIDTH_KEY, constrained);
      return constrained;
    });
    // same pattern for context width
  };

  constrainSavedWidths();
  window.addEventListener("resize", constrainSavedWidths);
  return () => window.removeEventListener("resize", constrainSavedWidths);
}, [getMaximumContextWidth, getMaximumNavigationWidth]);

Notice the desktop media check. On smaller screens the app still uses drawers; the resize persistence is not reapplied there.

The handle owns input, not layout

ColumnResizeHandle is a reusable separator. It measures the panel, calculates the next width, then calls onResize. It does not know about CSS variables or storage keys. That split keeps the component reusable and leaves app-specific persistence in App.

type ColumnResizeHandleProps = {
  controls: string;
  label: string;
  maxWidth: number | (() => number);
  minWidth: number;
  panelRef: RefObject<HTMLElement | null>;
  resizeDirection: "left" | "right";
  onReset: () => void;
  onResize: (width: number) => void;
};

Pointer path

const handlePointerDown = (event: PointerEvent<HTMLDivElement>) => {
  if (event.button !== 0) {
    return;
  }

  const width = measurePanel();
  dragStart.current = { pointerId: event.pointerId, width, x: event.clientX };
  event.currentTarget.setPointerCapture(event.pointerId);
  setActive(true);
  event.preventDefault();
};

const handlePointerMove = (event: PointerEvent<HTMLDivElement>) => {
  if (dragStart.current?.pointerId !== event.pointerId) {
    return;
  }

  const movement = event.clientX - dragStart.current.x;
  const direction = resizeDirection === "right" ? 1 : -1;
  resizeTo(dragStart.current.width + movement * direction);
};

The sign flip matters. Dragging the left edge of the writing partner to the right should reduce that column. Dragging the right edge of navigation to the right should grow it.

Keyboard path

if (event.key === "Home") {
  nextWidth = minWidth;
} else if (event.key === "End") {
  nextWidth = maximum;
} else if (event.key === "ArrowLeft") {
  nextWidth =
    measuredWidth + (resizeDirection === "left" ? KEYBOARD_STEP : -KEYBOARD_STEP);
} else if (event.key === "ArrowRight") {
  nextWidth =
    measuredWidth + (resizeDirection === "right" ? KEYBOARD_STEP : -KEYBOARD_STEP);
}

The handle advertises itself as a vertical separator and reports current bounds to assistive technology:

The grid moved out of Tailwind template strings

Before this change, App selected one of two Tailwind grid template strings depending on whether the writing partner had a selected detail entry. Now App always uses workspace-grid, optionally adds workspace-grid--detail, and passes custom properties when the user has saved widths.

const workspaceColumnStyles = {
  ...(navigationColumnWidth === null
    ? {}
    : { "--navigation-column-width": `${navigationColumnWidth}px` }),
  ...(contextColumnWidth === null
    ? {}
    : { "--context-column-width": `${contextColumnWidth}px` }),
} as CSSProperties;
<main
  ref={workspaceRef}
  aria-label="ScribeAI writing workspace"
  className={`workspace-grid grid ... ${
    inbox.selectedEntry ? "workspace-grid--detail" : ""
  }`}
  style={workspaceColumnStyles}
>
@media (min-width: 80rem) {
  .workspace-grid {
    --navigation-column-width: 248px;
    --context-column-width: 320px;

    grid-template-columns:
      var(--navigation-column-width)
      minmax(0, 1fr)
      var(--context-column-width);
  }

  .workspace-grid--detail {
    --context-column-width: clamp(30rem, 38vw, 40rem);
  }
}

That refactor is what makes resizing practical: React only needs to update custom properties, and the CSS grid recalculates the track sizes.

Things future maintainers should check first

Detail width at very large screens: At min-width: 96rem, .workspace-grid is declared after .workspace-grid--detail and has the same specificity. That means the later default --context-column-width: 360px can override the detail clamp unless an inline saved context width is present.

@media (min-width: 96rem) {
  .workspace-grid {
    --navigation-column-width: 280px;
    --context-column-width: 360px;
  }
}

The previous layout gave selected writing partner detail a clamp at both xl and 2xl. If wide detail panels still matter at 2xl, this cascade order needs attention.

Pause and check your model

1. When a user drags the writing partner handle to the right, should the context column grow or shrink?
It should shrink. The writing partner handle uses resizeDirection="left", so pointer movement is multiplied by -1. Positive movement to the right subtracts from the starting width.
2. Why does App calculate maximum widths instead of only relying on fixed MAX_NAVIGATION_WIDTH and MAX_CONTEXT_WIDTH constants?
Because the true maximum depends on the current workspace width and the opposite column. The calculations reserve MIN_EDITOR_WIDTH, so the editor track is not consumed by both side panels.
3. What does a saved width of null mean in this implementation?
null means "do not set an inline CSS variable; use the stylesheet defaults." Resetting a column writes this state by removing the localStorage item.
4. Where would you start debugging if selected writing partner details are unexpectedly narrow on a 2xl display?
Start in src/index.css. The @media (min-width: 96rem) .workspace-grid rule appears after .workspace-grid--detail and can override its context-width clamp because both selectors have equal specificity.
