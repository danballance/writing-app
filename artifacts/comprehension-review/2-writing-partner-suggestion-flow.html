Primary behaviour change

The app now has an agent loop around the editor

The previous TypeScript version was mostly layout: a BlockNote editor, a static context gutter, and a static layout-suggestions strip. This diff moves editor ownership up into App.tsx, adds a suggestion feed/inbox model, and renders one SuggestionDock on desktop or inside a wider mobile drawer.

The context source is revisioned

createAgentContextSource stores a snapshot and increments its revision only when the accepted block fingerprint changes.

updateDocument(blocks) {
  const nextFingerprint = fingerprint(blocks);
  if (nextFingerprint === currentFingerprint) {
    return;
  }

  currentFingerprint = nextFingerprint;
  snapshot = { revision: snapshot.revision + 1, blocks };
  listeners.forEach((listener) => listener(snapshot));
}

The inbox reducer is where feed events become UI state. The important behaviour is not just adding items. The reducer also protects selected and active-preview entries when a queue is trimmed or a suggestion is retracted.

// src/suggestions/inbox.ts
function enforceLimit(state: InboxState): InboxState {
  if (state.entries.length <= 30) return state;

  const protectedIds = new Set(
    [state.selectedId, state.activePreviewId].filter(
      (id): id is string => Boolean(id),
    ),
  );
  const evictionOrder = [...state.entries]
    .filter((entry) => !protectedIds.has(entry.item.id))
    .sort((a, b) => {
      if (a.viewed !== b.viewed) return a.viewed ? -1 : 1;
      return a.item.createdAt - b.item.createdAt;
    });

}

The inline preview is deliberately separate from accepted text

Preview is limited to text suggestions. App.handlePreview exits if another preview is active or the selected item is not a snippet, fact, or term. It also finds an insertion point among accepted blocks only, so existing preview blocks do not become anchors for new previews.

// src/App.tsx
const handlePreview = (item: SuggestionItem) => {
  if (inbox.activePreviewId || !isTextSuggestion(item)) {
    return;
  }

  const acceptedBlocks = editor.document.filter(
    (block) => block.type !== "suggestionPreview",
  );
  const referenceBlock =
    acceptedBlocks.find((block) => block.id === lastActiveBlockId) ??
    acceptedBlocks.at(-1);
  if (!referenceBlock) return;

  const preview = editor.insertBlocks(
    [{
      type: "suggestionPreview",
      props: { suggestionId: item.id, targetBlockId: referenceBlock.id },
      content: item.insertText,
    }],
    referenceBlock,
    "after",
  )[0];

  if (preview) {
    inbox.previewStarted(item.id);
    editor.setTextCursorPosition(preview.id, "end");
    ...
  }
};

The custom block in schema.tsx defines what happens when the user accepts or cancels. Accept replaces the preview block with a normal paragraph containing the edited preview content. Cancel removes the preview block and tries to restore the cursor to the original target block.

// src/editor/schema.tsx
const suggestionPreviewBlock = createReactBlockSpec(
  {
    type: "suggestionPreview",
    propSchema: {
      suggestionId: { default: "" },
      targetBlockId: { default: "" },
    },
    content: "inline",
  },
  {
    render: ({ block, editor, contentRef }) => {
      const resolve = (outcome: "accepted" | "cancelled") => {
        const fullEditor = editor as unknown as BlockNoteEditor;
        if (outcome === "accepted") {
          const { insertedBlocks } = fullEditor.replaceBlocks(
            [block.id],
            [{ type: "paragraph", content: block.content } as unknown as PartialBlock],
          );
          ...
        } else {
          fullEditor.removeBlocks([block.id]);
          ...
        }

        emitPreviewResolution({ suggestionId: block.props.suggestionId, outcome });
      };
      ...
    }
  }
);

What future maintainers should keep in their heads

Only one preview can be active. The dock disables preview when activePreviewId exists, and App.handlePreview also refuses to insert another. If a future code path inserts preview blocks directly, it must update the inbox state or this invariant breaks.
Preview resolution travels through a module-level event bus. schema.tsx cannot call the React inbox hook directly, so it emits through previewEvents.ts. The app subscribes in an effect. This is small and contained, but it depends on cleanup being correct when the app unmounts or hot reloads.
Accepted suggestions lose suggestion metadata. Accept replaces the preview with a plain paragraph. That is consistent with the current code, but future analytics or provenance features would need a different persistence path.
Mermaid changes bundle shape. The component uses import("mermaid"), but the production build still reported several large chunks and one large app chunk. Build passed, but this is a performance area to watch.

The test suite now covers the agent state model

Vitest is now configured with jsdom and a cleanup setup file. Four test files were added, and npm test passed with 6 tests.

  • SuggestionDock.test.tsx checks that the dock is one unified stream, not a tabbed gutter, and that text-detail view can request a preview.
  • inbox.test.ts checks dedupe, the 30-item cap, and stale/withdrawn handling when an active preview is updated and retracted.
  • contextSource.test.ts checks that unchanged block snapshots do not publish extra revisions.
  • mockSuggestionFeed.test.ts checks initial events, steering events, document-observation events, and timer cleanup after unsubscribe.
Coverage gap to know about: the custom BlockNote preview block's accept/cancel DOM behaviour is not directly tested here. That path is exercised by app behaviour manually or by build-time type checks, but there is no dedicated automated test for replacing a preview with a paragraph or restoring cursor position.

Pause here: can you reason through the new path?

1. A user clicks "Preview in document" on a mind-map suggestion. What should happen?

Nothing should be inserted. App.handlePreview calls isTextSuggestion, which only accepts snippet, fact, and term. The dock also only renders the preview button for text suggestions.

2. Why does getAcceptedDocumentBlocks skip suggestionPreview blocks?

Because preview text is tentative. The agent context should reflect accepted document content, not a draft suggestion the user may cancel or edit before accepting.

3. What would break if preview resolution did not call emitPreviewResolution?

The editor block would still be accepted or removed, but the inbox would keep activePreviewId set. That would block later previews and leave the suggestion entry in the wrong state.

4. When the queue exceeds 30 entries, which suggestions are protected from eviction?

The currently selected suggestion and the active-preview suggestion are protected. The reducer builds protectedIds from selectedId and activePreviewId before choosing old viewed entries for eviction.

5. Where would you start debugging if document-change suggestions stop appearing?

Start at App.handleEditorChange and contextSource.updateDocument(getAcceptedDocumentBlocks(editor)). Then check whether createMockSuggestionFeed subscribed to context.subscribeToDocument and whether the snapshot fingerprint is actually changing.
