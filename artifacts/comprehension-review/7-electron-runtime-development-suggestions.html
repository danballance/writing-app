Comprehension review

Electron is now the required runtime, and mock suggestions enter through the persisted desktop path.

This change removes the browser-only mock feed and makes development behave more like the packaged app: Vite still serves the renderer, but Electron owns the bridge, windows, storage, and suggestion delivery.

Runtime boundary

App no longer has a browser fallback. It receives a required DesktopBridge; missing preload produces a diagnostic page.

Development suggestions

The old BroadcastChannel mock feed is gone. A development menu opens an Electron window that writes suggestions through IPC and storage.

Build path

vite-plugin-electron replaces the hand-written desktop/build.mjs esbuild script and participates in npm run dev and npm run build.

The boundary that moved: browser page to Electron renderer

The most important interface change is small in code but broad in behavior. getDesktopBridge used to return undefined, and App selected either the Electron feed or the injected mock feed. Now missing window.scribe is an error handled before App is constructed.

// src/desktop/desktopClient.ts
export function getDesktopBridge(): DesktopBridge {
  const bridge = window.scribe;
  if (!bridge) {
    throw new Error("ScribeAI requires the Electron desktop runtime.");
  }
  return bridge;
}
// src/main.tsx
try {
  const desktop = getDesktopBridge();
  if (normalizedPath === MOCK_SUGGESTION_PATH) {
    const development = window.scribeDevelopment;
    rootView =
      MockSuggestionController && development ? (
        <Suspense fallback={null}>
          <MockSuggestionController
            createSuggestion={development.createSuggestion}
          />
        </Suspense>
      ) : (
        <RuntimeRequired message="The mock suggestion controller is only available from the Electron development menu." />
      );
  } else {
    rootView = <App desktop={desktop} />;
  }
} catch (error) {
  rootView = <RuntimeRequired message={...} />;
}

What to notice

RuntimeRequired is not a fallback runtime. It prevents the app from running without the preload bridge. That means a direct browser visit to the Vite URL no longer exercises the real application path.

The mock controller is also lazy-loaded only under import.meta.env.DEV, so production builds do not include the controller route as an enabled feature.

Inside App, the optional checks around desktop are removed. Hydration, autosave, provider configuration, source import, agent controls, and suggestion-state persistence now assume the bridge exists.

// src/App.tsx
type AppProps = {
  desktop: DesktopBridge;
};

export default function App({ desktop }: AppProps) {
  ...
  const feed = useMemo(() => createDesktopSuggestionFeed(desktop), [desktop]);
  const saveSuggestionState = useCallback(
    (state: PersistedSuggestionState) => {
      void desktop.saveSuggestionState(state);
    },
    [desktop],
  );
  ...
  const documentHydratedRef = useRef(false);
}

The invariant after this change: if App is mounted, Electron IPC must be available. Any future browser-only test harness or Storybook-like environment will need to provide a fake DesktopBridge rather than relying on optional behavior in App.

const developmentServerUrl = process.env.VITE_DEV_SERVER_URL;
const isDevelopment = Boolean(developmentServerUrl);
let mockSuggestionWindow: BrowserWindow | undefined;

if (isDevelopment) {
  ipcMain.handle(
    "scribe:development.suggestion.create",
    async (event, item: unknown) => {
      if (!validateSender(event.sender)) throw new Error("Unknown renderer");
      if (!isSuggestionItem(item)) {
        throw new Error("Invalid development suggestion");
      }
      return storage.call("development.suggestion.create", { item });
    },
  );
}
// desktop/main.ts
function openMockSuggestionWindow() {
  if (!developmentServerUrl) return;
  if (mockSuggestionWindow && !mockSuggestionWindow.isDestroyed()) {
    mockSuggestionWindow.focus();
    return;

// desktop/storage.ts
function createDevelopmentSuggestion(params: unknown) {
  const item = (params as { item?: unknown }).item;
  if (!isSuggestionItem(item)) {
    throw new Error("Invalid development suggestion");
  }
  return createSuggestion({
    item,
    expectedDocumentRevision: getDocument().revision,
  });
}

Risk worth carrying in your head

The development bridge is guarded by VITE_DEV_SERVER_URL in main and by --scribe-development in preload. During development, the workspace window also receives that argument, so any dev Electron renderer window with the exposed bridge can call the development IPC. The handler still validates shape and sender, and it is not registered in production. Treat it as a dev convenience, not a security boundary.

The validation code moved from mock-only to shared runtime code

The deleted src/dev/mockSuggestions/mockSuggestionValidation.ts logic is reintroduced as src/suggestions/validation.ts. That matters because the validator is now imported by three layers: mock draft building, Electron main, and storage.

// src/suggestions/validation.ts
export function isSuggestionItem(value: unknown): value is SuggestionItem {
  if (!isRecord(value)) return false;

  const hasCommonFields =
    isNonEmptyString(value.id) &&
    isNonEmptyString(value.dedupeKey) &&
    isNonEmptyString(value.title) &&
    isNonEmptyString(value.summary) &&
    isNonEmptyString(value.body) &&
    Array.isArray(value.sourceLabels) &&
    value.sourceLabels.every((label) => typeof label === "string") &&
    typeof value.createdAt === "number" &&
    Number.isFinite(value.createdAt);

  if (!hasCommonFields) return false;

  if (value.kind === "snippet" || value.kind === "fact" || value.kind === "term") {
    return isNonEmptyString(value.insertText);
  }

  if (value.kind === "outline" || value.kind === "layout") {
    return isStructureNodes(value.nodes);
  }

  if (value.kind === "mindMap") {
    return isNonEmptyString(value.mermaidSource) &&
      isNonEmptyString(value.accessibleDescription);
  }

  return false;
}

This is a runtime validator, not a schema migration. It checks the current SuggestionItem shape before IPC reaches storage. If a new suggestion kind or required field is added later, this file must change along with src/suggestions/types.ts, the mock draft builder, and any storage paths that accept unknown input.

The controller now reports persistence outcomes, not channel availability

MockSuggestionController no longer opens or closes a publisher. It receives a createSuggestion function from the bootstrap layer, disables repeat submission while IPC is pending, and distinguishes duplicate rejection from success.

// src/dev/mockSuggestions/MockSuggestionController.tsx
type MockSuggestionControllerProps = {
  createSuggestion: (
    item: SuggestionItem,
  ) => Promise<{ accepted: boolean }>;
};

const submitSuggestion = async (event: FormEvent<HTMLFormElement>) => {
  event.preventDefault();
  ...
  setSubmitting(true);
  setMessage(undefined);
  try {
    const suggestion = buildMockSuggestion(draft);
    const result = await createSuggestion(suggestion);
    if (!result.accepted) {
      throw new Error("The suggestion was rejected as a duplicate.");
    }
    formRef.current?.reset();
    setKind("snippet");
    setMessage({
      type: "success",
      text: `Added “${suggestion.title}” to the persisted suggestion inbox.`,
    });
  } catch (error) {
    ...
  } finally {
    setSubmitting(false);
  }
};

Build ownership moved into Vite

The old desktop/build.mjs called esbuild directly for main, storage, agent, and preload entries. The new config makes Vite own those Electron builds through vite-plugin-electron/simple.

// package.json
"scripts": {
  "dev": "vite",
  "build": "tsc -b && tsc -p tsconfig.desktop.json --noEmit && vite build",
  "desktop": "npm run build && electron ."
}
// vite.config.ts
plugins: [
  cleanElectronOutput(),
  react(),
  tailwindcss(),
  electron({
    main: {
      entry: {
        main: "desktop/main.ts",
        storage: "desktop/storage.ts",
        agent: "desktop/agent.ts",
      },
      vite: {
        plugins: [notBundle({ filter: /^(?![./])/ })],
      },
    },
    preload: {
      input: fileURLToPath(new URL("./desktop/preload.ts", import.meta.url)),
      vite: {
        build: {
          rolldownOptions: {
            output: { entryFileNames: "preload.cjs" },
          },
        },
      },
    },
  }),
]

The lockfile reflects the same intent: esbuild is removed as a direct dev dependency and vite-plugin-electron is added. The custom cleanElectronOutput plugin removes dist-electron at production build config resolution, so stale Electron entry outputs are less likely to survive a build.

Pause and check the model

These questions focus on behavior you would need while debugging or extending this change.

1. If someone opens the Vite dev URL directly in a browser, why does the full app no longer mount?

getDesktopBridge throws when window.scribe is absent. src/main.tsx catches that and renders RuntimeRequired instead of App. App now requires a DesktopBridge prop and no longer builds a mock feed fallback.

2. Where does a mock suggestion become durable?

In desktop/storage.ts, development.suggestion.create calls createDevelopmentSuggestion, which validates the item and then calls createSuggestion with expectedDocumentRevision: getDocument().revision. The storage test hydrates afterward and expects the item in the persisted suggestions projection.

3. What would break if src/suggestions/validation.ts were not included in tsconfig.desktop.json?

Electron's TypeScript project imports that validator from desktop/main.ts and desktop/storage.ts. Without including it, the desktop type-check could miss or reject the shared source dependency, depending on TypeScript project settings. The diff explicitly adds the file to the desktop project include list.

4. Why does the controller test inject a function instead of asserting on BroadcastChannel?

The controller no longer owns transport setup. Its boundary is now createSuggestion(item): Promise<{ accepted: boolean }>, supplied by the Electron development bridge at bootstrap. Channel publication and unsubscribe cleanup are obsolete because the channel files were deleted.

5. If a development suggestion is rejected as a duplicate, which layer reports that to the user?

Storage returns { accepted: false } through the development IPC path. MockSuggestionController checks that result and throws "The suggestion was rejected as a duplicate.", then renders it in an alert message.
