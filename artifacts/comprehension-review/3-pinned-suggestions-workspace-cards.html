Pinned suggestions now have a second life on the workspace.

This uncommitted change turns the suggestion inbox into a three-place system: the regular queue, a pinned list, and draggable workspace cards over the editor. The reducer owns the transitions; the dock exposes the controls; the editor renders the desktop canvas.

The new route through the app

Before this diff, src/suggestions/inbox.ts managed a single bounded queue of suggestions, plus selection and document preview state. The working tree adds two more locations for the same suggestion concept: pinned entries and workspace pins.

agent event->queue entry->pinned snapshot->workspace card->pinned snapshot

The important boundary is in src/App.tsx. App wires the reducer outputs into the two surfaces that need them: SuggestionDock and EditorWorkspace.

The reducer now protects three kinds of suggestion state

Most of the behavioural change is concentrated in src/suggestions/inbox.ts. The old state had entries, selectedId, and activePreviewId. The new state adds pinned snapshots, workspace geometry, and a monotonically increasing z-index counter. 

// src/suggestions/inbox.ts
export type PinnedInboxEntry = InboxEntry & {
  pinnedAt: number;
};

export type WorkspacePin = WorkspacePinRect & {
  item: SuggestionItem;
  pinnedAt: number;
  pendingInitialPlacement: boolean;
  zIndex: number;
};

type InboxState = {
  entries: InboxEntry[];
  pinnedEntries: PinnedInboxEntry[];
  workspacePins: WorkspacePin[];
  selectedId?: string;
  activePreviewId?: string;
  nextZIndex: number;
  status: AgentStatus;
};
Queue

Updated by agent events, sorted unread-first, capped at 30 entries.

Pins

Copied snapshots, sorted newest pin first, excluded from the 30-entry queue cap.

Workspace

Pinned snapshots with rectangle and z-index. Hidden until initial placement is committed.

Pinning freezes the item

Pinning is not a reference to the live queue entry. The reducer removes the item from entries and stores a JSON copy in pinnedEntries. That supports the test named "freezes pinned snapshots and excludes them from the queue limit". 

// src/suggestions/inbox.ts
function copySuggestion(item: SuggestionItem): SuggestionItem {
  return JSON.parse(JSON.stringify(item)) as SuggestionItem;
}

if (action.type === "pin") {
  const entry = state.entries.find((candidate) => candidate.item.id === action.id);
  if (!entry) {
    return state;
  }
  return {
    ...state,
    entries: state.entries.filter((candidate) => candidate.item.id !== action.id),
    pinnedEntries: [
      ...state.pinnedEntries,
      {
        ...entry,
        item: copySuggestion(entry.item),
        stale: false,
        withdrawn: false,
        pinnedAt: action.pinnedAt,
      },
    ],
  };
}

A later suggestion.updated maps only over state.entries, and a retraction returns early when the id is pinned or already on the workspace.

Workspace transitions are explicit

Moving to the workspace only works from the pinned list. The reducer does not search the regular queue for workspace.place. It removes the pinned entry, clears selection if needed, and creates a WorkspacePin with pendingInitialPlacement: true. 

// src/suggestions/inbox.ts
if (action.type === "workspace.place") {
  if (state.activePreviewId === action.id) {
    return state;
  }
  const entry = state.pinnedEntries.find(
    (candidate) => candidate.item.id === action.id,
  );
  if (!entry) {
    return state;
  }
  return {
    ...state,
    selectedId: state.selectedId === action.id ? undefined : state.selectedId,
    pinnedEntries: state.pinnedEntries.filter(
      (candidate) => candidate.item.id !== action.id,
    ),
    workspacePins: [
      ...state.workspacePins,
      {
        item: entry.item,
        pinnedAt: entry.pinnedAt,
        pendingInitialPlacement: true,
        ...action.rect,
        zIndex: state.nextZIndex,
      },
    ],
    nextZIndex: state.nextZIndex + 1,
  };
}

Returning from workspace recreates a pinned entry and marks it viewed. Geometry changes also flip pendingInitialPlacement off, which is what allows the card to render. 

// src/suggestions/inbox.ts
if (action.type === "workspace.geometry") {
  return {
    ...state,
    workspacePins: state.workspacePins.map((pin) =>
      pin.item.id === action.id
        ? { ...pin, ...action.rect, pendingInitialPlacement: false }
        : pin,
    ),
  };
}

The editor becomes a measured canvas

The old DocumentEditor rendered a centered BlockNote editor inside the scroll area. The new version wraps that editor in a full-width relative canvas and overlays WorkspacePins on top.

Initial placement is a two-step process. The reducer creates a pending pin. Then DocumentEditor waits for a frame, measures the scroll area and canvas, computes a cascaded rectangle near the visible right side, and commits it through onWorkspacePinGeometryChange. 

// src/components/DocumentEditor.tsx
const getInitialWorkspacePinRect = useCallback(
  (pin: WorkspacePin, stackIndex: number) => {
    const scroll = scrollRef.current;
    const canvas = canvasRef.current;
    if (!scroll || !canvas) {
      return undefined;
    }
    const baseSize = getInitialWorkspacePinSize(pin.item);
    const width = Math.min(baseSize.width, Math.max(280, canvas.clientWidth - 32));
    const height = Math.min(baseSize.height, Math.max(180, canvas.clientHeight - 32));
    const cascade = (stackIndex % 5) * 24;
    const scrollRect = scroll.getBoundingClientRect();
    const canvasRect = canvas.getBoundingClientRect();
    const visibleTop = Math.max(0, scrollRect.top - canvasRect.top);
    return {
      x: Math.max(16, canvas.clientWidth - width - 24 - cascade),
      y: Math.min(
        Math.max(16, visibleTop + 24 + cascade),
        Math.max(16, canvas.clientHeight - height - 16),
      ),
      width,
      height,
    };
  },
  [],
);
Workspace pin cardmove | return

Card content: badge, summary, body, and visual content for outline/layout/mind-map suggestions.

Geometry controls: pointer drag, pointer resize, keyboard arrow movement, keyboard arrow resizing.

Bounds: cards are clamped to the measured canvas with a minimum size of 280 by 180 and 16px edge padding.

// src/components/WorkspacePins.tsx
function clampRect(
  rect: WorkspacePinRect,
  bounds: { width: number; height: number },
): WorkspacePinRect {
  const maxWidth = Math.max(MIN_WIDTH, bounds.width - EDGE_PADDING * 2);
  const maxHeight = Math.max(MIN_HEIGHT, bounds.height - EDGE_PADDING * 2);
  const width = clamp(rect.width, MIN_WIDTH, maxWidth);
  const height = clamp(rect.height, MIN_HEIGHT, maxHeight);
  return {
    x: clamp(rect.x, EDGE_PADDING, bounds.width - width - EDGE_PADDING),
    y: clamp(rect.y, EDGE_PADDING, bounds.height - height - EDGE_PADDING),
    width,
    height,
  };
}

The workspace overlay itself is desktop-only: className="pointer-events-none absolute inset-0 z-20 hidden xl:block". Cards opt back into interaction with pointer-events-auto on the card. That means the overlay should not block normal editor clicks outside cards.

Assumptions a future maintainer should check before extending this

Pins use JSON cloning.

copySuggestion is safe for the current SuggestionItem union because the types are plain data. If suggestions later carry Dates, class instances, Maps, or functions, the snapshot behaviour will change or lose data.

Workspace pins are desktop-only in the UI.

Once a pin is placed on the workspace, it is removed from pinnedEntries. On non-xl layouts, the workspace overlay is hidden, so there is no visible return control in the changed code.

Agent updates do not refresh pins.

This is probably intentional, but it means a pinned suggestion can become outdated relative to the feed. The code preserves the snapshot and ignores retractions for pinned/workspace items.

Z-index only increases.

nextZIndex increments on placement and raise. That is simple and predictable for a browser session, but there is no compaction or persistence strategy in this diff.

Pause here and check your mental model

Questions

1. A suggestion is pinned, then the agent sends suggestion.updated for the same id. Which copy changes?

The pinned copy does not change. The reducer's update branch maps over state.entries only, and pinning stores copySuggestion(entry.item) in pinnedEntries. The queue copy is gone after pinning, so the pinned snapshot remains as it was.

2. Why does a workspace pin start with pendingInitialPlacement: true?

The reducer can create a pin before the editor canvas has been measured. DocumentEditor computes a better visible rectangle on the next animation frame, commits that geometry, and the geometry reducer sets pendingInitialPlacement to false. WorkspacePins filters out pending pins, so users should see the card after placement is measured.

3. What would break if SuggestionVisual stayed private to SuggestionDock?

The workspace card would either have to duplicate outline/layout/mind-map rendering or omit those visuals. The extraction lets both detail view and workspace cards render structured suggestions through the same badge/tree/mermaid path.

4. Where would you start debugging if a dragged card jumps outside the editor?

Start in WorkspacePins.tsx, especially clampRect, the measured bounds from canvas.clientWidth/clientHeight, and the pointer operation delta calculation. If the stored geometry is valid but rendering is wrong, check the canvas wrapper in DocumentEditor.tsx.

5. Which invariant keeps a workspace card above another card after the user interacts with it?

workspace.raise assigns the selected pin state.nextZIndex and then increments nextZIndex. The card style uses zIndex: pin.zIndex, so the raised card renders above cards with older z-index values.
