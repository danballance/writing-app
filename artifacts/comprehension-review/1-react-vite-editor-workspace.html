working tree review

Static mockup becomes a Vite React writing workspace

The committed project started as a single HTML/CSS mockup. The current working tree turns it into a typed React app with Tailwind v4 styling, a BlockNote editor, componentized side panels, and mobile drawers for navigation and AI context.

History anchor Only one commit exists: ec06c0b Initial commit with html mock up. There is no multi-commit design trail, so behavioural intent here is inferred from the diff and the code that replaced the mockup.

The change at the boundary

The app boundary changed from browser-rendered markup and a linked stylesheet to a JavaScript application mounted into #root. That is the largest behavioural shift: UI that used to be fixed markup can now hold state, respond to viewport changes, and host an editable document surface.

Before

The initial commit rendered the whole workspace directly in index.html and loaded styles.css. No app runtime or package manifest existed.

Follow a user action through the new runtime

A good way to understand the new app is to follow the mobile drawer path. On desktop, the side panels are normal grid columns. Below the xl breakpoint, the header buttons open modal drawers. This is where most of the new control flow lives.

1. Header buttonDocumentHeader receives callbacks and calls them from responsive-only icon buttons.
2. App statenavigationOpen or contextOpen flips to true.
3. Drawer mountsResponsiveDrawer returns modal markup only when open is true.
4. Focus loopEscape closes the drawer. Tab wraps between focusable elements in the panel.
5. Desktop resetA media query closes drawers when the viewport reaches desktop width.
export default function App() {
  const [navigationOpen, setNavigationOpen] = useState(false);
  const [contextOpen, setContextOpen] = useState(false);

  useEffect(() => {
    const desktopQuery = window.matchMedia("(min-width: 1280px)");
    const closeDrawersAtDesktop = (event: MediaQueryListEvent) => {
      if (event.matches) {
        setNavigationOpen(false);
        setContextOpen(false);
      }
    };

    desktopQuery.addEventListener("change", closeDrawersAtDesktop);
    return () => desktopQuery.removeEventListener("change", closeDrawersAtDesktop);
  }, []);

  return (
    <main className="grid h-dvh ... xl:grid-cols-[248px_minmax(0,1fr)_320px] ...">
      <div className="hidden min-h-0 xl:block"><Sidebar /></div>
      <EditorWorkspace
        onOpenNavigation={() => setNavigationOpen(true)}
        onOpenContext={() => setContextOpen(true)}
      />
      <div className="hidden min-h-0 xl:block"><ContextGutter /></div>
    </main>
  );
}

The responsive behaviour is not only CSS. The media-query effect prevents an open mobile drawer from lingering after a resize to desktop, where the same content is already visible in grid columns.

The most stateful code is the drawer

ResponsiveDrawer is the part of the change with the most behaviour beyond rendering. It mounts a dialog, focuses the close button, closes on Escape, closes on backdrop click, wraps Tab within the panel, and restores focus after cleanup.

useEffect(() => {
  if (!open) {
    return;
  }

  const previouslyFocused = document.activeElement as HTMLElement | null;
  const focusFrame = window.requestAnimationFrame(() => closeButtonRef.current?.focus());
  const handleKeyDown = (event: KeyboardEvent) => {
    if (event.key === "Escape") {
      event.preventDefault();
      onClose();
      return;
    }

    if (event.key !== "Tab") {
      return;
    }

    const focusableElements = panelRef.current?.querySelectorAll<HTMLElement>(
      'button:not([disabled]), input:not([disabled]), [href], [tabindex]:not([tabindex="-1"])',
    );

    if (!focusableElements?.length) {
      return;
    }

    const firstElement = focusableElements[0];
    const lastElement = focusableElements[focusableElements.length - 1];

    if (event.shiftKey && document.activeElement === firstElement) {
      event.preventDefault();
      lastElement.focus();
    } else if (!event.shiftKey && document.activeElement === lastElement) {
      event.preventDefault();
      firstElement.focus();
    }
  };

  document.addEventListener("keydown", handleKeyDown);

  return () => {
    window.cancelAnimationFrame(focusFrame);
    document.removeEventListener("keydown", handleKeyDown);
    window.requestAnimationFrame(() => previouslyFocused?.focus());
  };
}, [onClose, open]);
Maintainer note: onClose is passed as an inline arrow from App. If future parent state updates happen while a drawer is open, this effect can clean up and re-run because onClose has a new identity. In the current code, user edits in BlockNote should not re-render App, so this is mainly a future-change risk. If drawer focus starts jumping after more state is added, wrapping the close handlers in useCallback would be a reasonable first check.

Styling moved from authored CSS classes to Tailwind plus targeted BlockNote CSS

The deleted styles.css held custom selectors for every part of the mockup. The new src/index.css sets Tailwind v4 up, defines theme variables, applies base accessibility rules, and then customizes BlockNote using the data-editor-surface marker from DocumentEditor.

The @source directive matters because the project is using Tailwind v4 and BlockNote's shadcn package. It tells Tailwind to include class usage from that dependency. The rest of the app mostly uses utility classes directly in JSX.

What future you should be careful about

Do not assume all visible controls are wired

The header's Export, Share, history, generate ideas, and more buttons have no action handlers. Context snippet copy/insert buttons also render without behaviour. The UI looks app-like, but the implemented behaviours are narrower.

Keep editor styling coupled to data-editor-surface

The BlockNote CSS selectors rely on data-editor-surface. If that attribute is removed from BlockNoteView, editor typography and colors will stop matching the workspace styling.

Drawer accessibility depends on mounted content

The focus loop queries buttons, inputs, links, and non-negative tabindex elements inside the panel. If a future drawer has no focusable children, the loop returns early. This code is safe, but the resulting dialog would not have a meaningful tab target beyond any close button that remains.

Large client bundle is now part of the baseline

The production build succeeded but reported a large chunk. If startup performance becomes a goal, the editor package is a likely place to inspect for code-splitting.

Comprehension check

Use these to test whether the main paths are clear before editing this code.

1. On a phone-sized viewport, what path opens the navigation panel?

DocumentHeader renders a responsive-only menu IconButton. Its onClick calls onOpenNavigation, passed from EditorWorkspace props into DocumentHeader. In App, that callback sets navigationOpen to true, causing the left ResponsiveDrawer to mount with Sidebar as its children.

2. What would break visually if data-editor-surface were removed from BlockNoteView?

The custom BlockNote selectors in src/index.css would no longer match. The editor would still render, but the configured editor colors, transparent background, heading style, paragraph font, spacing, and mobile padding rules would not apply through those selectors.

3. Which parts of the app are real interactions now, and which are placeholders?

Real interactions in this diff: BlockNote editing and responsive drawer open/close, including Escape, backdrop click, focus movement, and desktop resize cleanup. Placeholder controls include Export, Share, history, generate ideas, more options, context tab switching, snippet copy, snippet insert, source upload, and navigation buttons.

4. Why does the App media-query effect close both drawers at min-width: 1280px?

At xl, the layout shows Sidebar and ContextGutter as grid columns. Closing the drawer state during the transition prevents a mobile modal version of the same content from staying mounted on top of the desktop layout.

5. Where would you start debugging if the editor rendered but looked unstyled?

Start with DocumentEditor.tsx and src/index.css: confirm BlockNoteView imports @blocknote/shadcn/style.css, still has data-editor-surface, and that Tailwind is processing src/index.css through the Vite Tailwind plugin. The project also uses @source "../node_modules/@blocknote/shadcn", which is relevant to generated classes from that package.
